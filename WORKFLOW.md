# WORKFLOW.md — процесс разработки AquaGeo

Платформа: GitHub через `gh`. Слово MR в этом документе означает Pull Request.
Если проект на GitLab, меняются только CI-файл и команды (`glab` вместо `gh`),
сам процесс тот же.

## Ветки

- `main`: защищённая, всегда стабильная и зелёная. Прямые пуши запрещены.
- `develop`: интеграционная ветка, сюда сливается работа по задачам.
- `feature/<issue#>-<slug>`: ветка под одну issue, отводится от `develop`.

## Процесс по каждой issue

1. Отвести `feature/<issue#>-<slug>` от свежего `develop`.
2. Реализовать задачу и тесты. Коммиты атомарные, conventional commits.
3. Открыть MR `feature → develop`. CI (линт + тесты) должен быть зелёным,
   затем слить в `develop`. Это шаг «сначала пушим в develop».
4. Открыть MR `develop → main` с полным описанием по шаблону ниже. Прогоняются
   CI и автотесты. Когда всё зелёное, MR принимается автоматически
   (`gh pr merge --auto`). Красное не мёржим никогда.
5. Issue закрывается через `Closes #N` в описании MR.

Промоут `develop → main` делаем по каждой issue, чтобы `main` всегда был
актуален и каждый MR был протестирован. Если захочется меньше MR, промоут можно
собирать пачкой в конце каждого milestone, но по умолчанию per issue.

## Нейминг MR

- Совпадает с названием issue. Если MR решает одну issue:
  `Resolve "<название issue>"`.
- Если по ходу работы цель сместилась, заголовок обновляется в стиле
  «Русский текст + English keywords».

## Описание MR (шаблон)

Формат «технического резюме», профессиональный отчёт о работе. Markdown с
иерархией `#` и `##`.

```
Closes #<issue>        (по одной строке на каждую issue, если MR создан по issue)

# Что сделано
Перечисление изменений по слоям или модулям.

# Основные изменения
Самый большой блок. Что и как изменено и какие последствия. Сухая выжимка без
оправданий. При большом диффе разбить на подразделы ##.

# Зачем это сделано
Аргументация цели изменений.

# Тестирование
Отчёт о покрытии (unit, integration, нагрузочные). Пишется только если в диффе
менялись тесты.
```

## CI/CD (GitHub Actions)

- На MR в `develop` и в `main`: джоба backend (линт ruff + pytest с сервисом
  postgis) и джоба frontend (npm ci + build + vitest).
- `main` защищён: для merge требуются зелёные проверки CI, плюс включён
  auto-merge.
- Claude Code включает auto-merge на MR `develop → main` и дожидается зелёного
  CI, затем MR вливается сам.

## Коммиты

Conventional commits: `feat:`, `fix:`, `chore:`, `docs:`, `test:`, `refactor:`.
Заголовок короткий на английском, в теле можно по-русски.

## Конфиг и секреты

LLM-ключ Google AI Studio лежит в `.env` как `GEMINI_API_KEY`. Не коммитить.
litellm читает `GEMINI_API_KEY` из окружения сам для моделей `gemini/*`
(ключ в коде руками не передаём). `ANTHROPIC_API_KEY` — опциональная
альтернатива. В `.env.example` держать актуальные плейсхолдеры. Реальные
значения только локально и в секретах CI.
